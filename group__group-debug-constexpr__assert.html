Detailed Description

An assert-like macro with well-defined behavior for compile-time evaluation.

Remarks
You might ask "why does this exist? Just use <tt>assert</tt>!", but I promise you that I'm not crazy. Well, maybe a little, but that's not important right now. Calling non-constexpr functions at compile-time is "ill-formed, no diagnostic required". Nearly all implementations of assert will call an implementation-defined built-in function if the assertion fails. This means that the compiler isn't required to stop compilation and emit an error when assert(false); is evaluated at compile-time. Of couirse, will nearly all compilers fail and emit an error? Yes! Do they have to? No. Hence, why this exists. Unlike assert(false), a compile-time evaluated constexpr_assert(false) guarentees compilation failure and should cause the compiler to emit a relevant diagnostic message.

#define FGL_DEBUG_CONSTEXPR_ASSERT (   assertion)

An assert-like macro with well-defined behavior for compile-time evaluation.

If constexpr_assert is evaluated at runtime, the expression is evaluated within a standard assert, but if it's evaluated at compile-time and the expression is resolves to false, then a compile-time error will be generated via an empty throw (which is explicitly illegal in a compile-time context).

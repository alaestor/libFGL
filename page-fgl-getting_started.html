Installing and using libFGL

With Conan

  1. If you don't have it already, install the Conan package manager
  2. Enable revisions by adding revisions_enabled=1 under [general] in your conan.conf file (~/.conan/conan.conf on Linux, userprofile%/.conan/conan.conf on Windows).
  3. Add the libFGL remote by running conan remote add fgl-conan https://fgl.jfrog.io/artifactory/api/conan/fgl-conan

  4. Add libFGL as a conan dependency

    It's recommended to always use the latest version of libFGL by requiring the latest alias

    [requires]
    libFGL/latest@alaestor/stable

    Specific versions can be required by following the format branch-commit (e.g. main-0123456789abcdef). Only the main branch is automatically published when a new commit is pushed, but others may occasionally be manually packaged.

    Note

    Some commits may not be available, including older commits before conan was supported.

    You can query the list of available versions by running conan search –remote=fgl-conan (it may take a while).

    Attention

    A "live at head" approach is strongly recommended: don't pin to a specific commit without good reason. Breaking changes are a feature! When something breaks, it's usually for a good reason.

    Only use versions from the main branch on the stable channel. Usage of versions from branches other than main , and versions released on channels other than @alaestor/stable are strongly discouraged.

    Versions on the @alaestor/volatile channel are subject to random revisions and deletions without warning and should be viewed as "temporary"! They should avoided unless explicitly directed by a maintainer (for example, when being used temporarily as a trouble-shooting measure).

Without Conan

  1. Download the libFGL source to where ever you want to install it, either by running git clone https://github.com/alaestor/libFGL.git or by downloading it as a ZIP archive from Github
  2. Add the library include path when you compile.
    • For Clang and GCC pass -I"your_path_to/libFGL/include" when compiling.
    • For MSVC pass /I"your_path_to/libFGL/include" when compiling.
    Replace your_path_to with the path to where you installed libFGL.

Hello World

// main.cpp
#include <array>
#include <FGLM> // all of libFGL + opt-in features
//#include <fgl.hpp> // all of libFGL without opt-in features
/// or you can be more selective:
// #define FGL_SHORT_MACROS
// #include <fgl/debug/constexpr_assert.hpp> // constexpr_assert
// #include <fgl/debug/output.hpp> // debug::output
// #include <fgl/types/range_constraints.hpp> // sized_range_same_as
// #include <fgl/utility/zip.hpp> // czip
/// silly example, implemented using fgl::zip & range constraints
constexpr int sum_int_pairs(
const fgl::sized_range_same_as<int> auto& a,
const fgl::sized_range_same_as<int> auto& b)
{
int sum{ 0 };
for (const auto& [av, bv] : fgl::czip(a, b))
sum += av + bv;
return sum;
}
int main()
{
[]() consteval
{
constexpr std::array a{ 1, 1, 1 }; // 3 elements
std::array b{ 1, 1, 1, 1 }; // 4 elements, last will be ignored
constexpr_assert(sum_int_pairs(a, b) == 6);
}();
fgl::debug::output("Hello, world!");
}

Example compiler arguments:

g++ -std=c++20 -I"your_path_to/libFGL/include" main.cpp -o main.exe

Output:

[GENERIC] Hello, world!

libFGL extensions will often require more elaborate dependencies and compilation steps. Refer to the extension's documentation.

Note

Depending on how your compiler arguments and environment configuration, you may need to include libFGL components with either "" or <> include syntax.

Other than for testing, libFGL doesn't support any particular build systems, package systems, or IDEs. You're on your own.

Compiling and Running Tests

libFGL is header-only, so it doesn't require compilation or linking. Just include the components you want and you're done. However, you should probably compile and run the unit tests to make sure everything's behaving properly in your environment.

Remove the .DEFAULT extension from tup.config.DEFAULT. This file contains system-specific paths and user preferences.

Use the tup build-system by executing tup in the directory that contains Tupfile.ini. The top-level Tupfile is configured to build and run all tests. If tup completes without warnings or errors, then the library should be fully-functional on the platform in question.
